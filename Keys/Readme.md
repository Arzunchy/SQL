## 📖 Overview

In DBMS, **keys are attributes (or sets of attributes)** used to:

* Uniquely identify records
* Maintain data integrity
* Establish relationships between tables
* Optimize queries

---

# 🔑 1. Primary Key

### Definition

Uniquely identifies each row in a table.

### Use Case (Understandable)

Use when you need **one main column to identify each record**.
For example, in a student table, `student_id` ensures:

* No two students have the same ID
* You can safely update/delete specific records

### Syntax

```sql
CREATE TABLE students (
    student_id INT,
    name VARCHAR(100),
    PRIMARY KEY (student_id)
);
```

### Rules

* Must be unique
* Cannot be NULL
* Only one per table

---

# 🔑 2. Candidate Key

### Definition

Minimal set of attributes that can uniquely identify a row.

### Use Case (Understandable)

Use when a table has **multiple possible unique identifiers**.
Example:

* `emp_id` → system ID
* `email` → also unique

Both can identify a row → both are candidate keys.

### Syntax

```sql
CREATE TABLE employees_ck (
    emp_id VARCHAR(10),
    email VARCHAR(100) NOT NULL,

    PRIMARY KEY (emp_id),
    UNIQUE (email)
);
```

### Rules

* Must be unique
* Must be minimal
* No NULL values

---

# 🔑 3. Super Key

### Definition

Any combination of columns that uniquely identifies a row.

### Use Case (Understandable)

Used to understand that:

* Even **extra columns + unique column** still give uniqueness

Example:

* `emp_id` → enough
* `emp_id + name` → still unique (but unnecessary)

### Syntax

```sql
-- Example usage (conceptual)
SELECT * FROM employees_ck
WHERE emp_id = 'E001' AND email = 'amit@corp.com';
```

### Rules

* May contain extra columns
* Candidate key ⊂ Super key

---

# 🔑 4. Alternate Key

### Definition

Candidate keys not selected as primary key.

### Use Case (Understandable)

Use when you want to **keep other unique fields safe from duplicates**
even if they are not primary.

Example:

* account_no → primary key
* pan_number → must also be unique

### Syntax

```sql
CREATE TABLE bank_accounts (
    account_no VARCHAR(10),
    pan_number VARCHAR(10) NOT NULL,

    PRIMARY KEY (account_no),
    UNIQUE (pan_number)
);
```

### Rules

* Unique and NOT NULL
* Could act as primary key

---

# 🔑 5. Foreign Key

### Definition

Links one table to another via primary key.

### Use Case (Understandable)

Use when data is **related across tables**.

Example:

* departments table → stores department info
* employees table → stores employees

Instead of repeating department data, you:

* store `dept_id` in employees
* link it to departments table

### Syntax

```sql
CREATE TABLE departments (
    dept_id VARCHAR(5),
    PRIMARY KEY (dept_id)
);

CREATE TABLE employees_fk (
    emp_id VARCHAR(10),
    dept_id VARCHAR(5),

    PRIMARY KEY (emp_id),
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);
```

### Rules

* Must match parent table values
* Can repeat
* Ensures referential integrity

---

# 🔑 6. Composite Key

### Definition

Combination of multiple columns forming a primary key.

### Use Case (Understandable)

Use when **one column is not enough** to uniquely identify data.

Example:

* A student can take multiple subjects
* A subject has many students

So:

* `(student_id, subject_id)` together = unique

### Syntax

```sql
CREATE TABLE attendance (
    student_id INT,
    subject_id INT,

    PRIMARY KEY (student_id, subject_id)
);
```

### Rules

* Combination must be unique
* Individual columns can repeat

---

# 🔑 7. Surrogate Key

### Definition

Artificial key generated by system.

### Use Case (Understandable)

Use when:

* No natural unique field exists
* OR natural key is too long/complex

Example:

* Instead of using email as ID → use auto ID

### Syntax

```sql
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT,
    product VARCHAR(100),

    PRIMARY KEY (order_id)
);
```

### Rules

* No real-world meaning
* Auto-generated
* Always unique

---

# 🔑 8. Natural Key

### Definition

Real-world meaningful attribute used as key.

### Use Case (Understandable)

Use when real-world data is already unique.

Example:

* Aadhaar number
* Email

No need to create extra ID.

### Syntax

```sql
CREATE TABLE citizens (
    aadhaar_no CHAR(14),
    name VARCHAR(100),

    PRIMARY KEY (aadhaar_no)
);
```

### Rules

* Has business meaning
* May change in real life

---

# 🔑 9. Unique Key

### Definition

Ensures all values in a column are unique.

### Use Case (Understandable)

Use when:

* Column must not have duplicates
* But is NOT the main identifier

Example:

* username must be unique
* but user_id is primary key

### Syntax

```sql
CREATE TABLE users (
    user_id INT,
    email VARCHAR(100),

    PRIMARY KEY (user_id),
    UNIQUE (email)
);
```

### Rules

* No duplicates allowed
* Allows one NULL (DB dependent)

---

# 🔑 10. Partial Key

### Definition

Attribute of weak entity requiring another key.

### Use Case (Understandable)

Use when a record **depends on another table**.

Example:

* An employee can have multiple dependents
* `dep_no` is unique only per employee

So:

* emp_id + dep_no → unique

### Syntax

```sql
CREATE TABLE dependents (
    emp_id VARCHAR(10),
    dep_no INT,

    PRIMARY KEY (emp_id, dep_no)
);
```

### Rules

* Not unique alone
* Works with parent key

---

# 🎁 Bonus: Secondary Key

### Definition

Used for searching, not unique.

### Use Case (Understandable)

Use when you frequently search/filter data.

Example:

* Searching employees by department

Adding index makes queries faster.

### Syntax

```sql
CREATE TABLE employees_sk (
    emp_id INT,
    dept VARCHAR(50),

    PRIMARY KEY (emp_id),
    INDEX idx_dept (dept)
);
```

### Rules

* Not unique
* Used for filtering
* Improves performance
